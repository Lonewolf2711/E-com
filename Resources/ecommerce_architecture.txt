ECOMMERCE CMS
Complete System Architecture
PHP 8  ·  MySQL  ·  Bootstrap 5  ·  Hostinger
Stack: PHP 8 · MySQL · Bootstrap 5 · Electro (frontend) · Mazer (admin) · Shared Hosting (Hostinger)

 
1. System Overview
This ecommerce CMS is a fully custom PHP 8 application built for shared hosting environments. It is designed as a reusable, deployable storefront and admin system covering the full commerce lifecycle from product management to order delivery tracking.

Property	Detail
Purpose	Reusable ecommerce CMS for freelance deployments
Backend	PHP 8 (procedural/MVC-lite, no framework dependency)
Database	MySQL 8 via PDO with prepared statements
Frontend	Electro Bootstrap 1.0 — HTML/CSS/JS ecommerce template
Admin	Mazer Admin Bootstrap — pure HTML/CSS/JS admin dashboard
Hosting	Shared hosting — Hostinger compatible (PHP + MySQL via cPanel)
Payments	Razorpay, Stripe, PayPal, UPI, Cash on Delivery (modular)
SEO	Clean slug-based URLs via .htaccess mod_rewrite
Security	PDO prepared statements, CSRF tokens, session auth, bcrypt

7. Payment Gateway Architecture
The payment system is modular. Each gateway lives in its own folder under payments/. A common interface ensures consistent order status updates regardless of which gateway is used.

7.1 Payment Flow
Cart → Checkout → Select Payment Method
  ├── Razorpay  → create_order.php → JS popup → verify_payment.php → update order
  ├── Stripe    → create_session.php → redirect to Stripe → webhook.php → update order
  ├── PayPal    → redirect to PayPal → callback.php → update order
  ├── UPI       → show UPI QR / deep link → manual confirmation in admin
  └── COD       → cod/handler.php → order created with payment_status=pending

7.2 Razorpay Integration Flow
1.	Customer clicks "Place Order" on checkout
2.	checkout.php calls payments/razorpay/create_order.php via AJAX
3.	create_order.php creates order in DB (status=pending) and calls Razorpay API to get rzp_order_id
4.	Frontend opens Razorpay modal with rzp_order_id
5.	Customer completes payment in modal
6.	Frontend sends razorpay_payment_id + razorpay_signature to verify_payment.php
7.	verify_payment.php validates HMAC signature, updates order status to paid, deducts stock
8.	Customer redirected to /order-success/{id}

8. Order Tracking System
Every order has a timeline. Admin adds tracking updates; customers see a visual timeline on the track-order page.

8.1 Tracking Statuses (in sequence)
Status	Meaning
Pending	Order placed, awaiting payment confirmation
Confirmed	Payment received, order confirmed by admin
Packed	Items packed and ready for dispatch
Shipped	Handed to courier, tracking number assigned
Delivered	Customer received the order
Cancelled	Order cancelled (shown in red on timeline)

8.2 Admin: Adding a Tracking Update
// admin/modules/order/tracking.php
$order_id = (int)$_POST["order_id"];
$status   = sanitize($_POST["status"]);
$message  = sanitize($_POST["message"]);

$stmt = $pdo->prepare(
  "INSERT INTO order_tracking (order_id, status, message, created_by)
   VALUES (?, ?, ?, ?)"
);
$stmt->execute([$order_id, $status, $message, $_SESSION["admin_id"]]);

// Also update orders.status
$pdo->prepare("UPDATE orders SET status=? WHERE id=?")
   ->execute([$status, $order_id]);

8.3 Customer: Track Order Page
Customer enters order number on /track-order. System retrieves all order_tracking rows for that order and renders them as a vertical timeline (Mazer's timeline component). Each entry shows status badge, message, and timestamp.

 
9. Security Architecture

Threat	Mitigation
SQL Injection	All DB queries use PDO prepared statements — no raw user input in SQL
XSS	All output escaped via htmlspecialchars() in a custom e() helper function
CSRF	All POST forms include a CSRF token (generated in session, validated on submit)
Password storage	bcrypt via password_hash(PASSWORD_BCRYPT) with cost factor 12
Session security	session.cookie_httponly = 1, session.cookie_secure = 1 (HTTPS), session.cookie_samesite = Strict
File uploads	Extension whitelist (jpg/png/webp only), MIME type check, random filename, stored outside webroot
Admin access	auth_check.php verifies session + role=admin at top of every admin file
API keys	Stored as environment variables or in config.php outside public root
Directory listing	Options -Indexes in .htaccess to prevent directory browsing
Error display	display_errors = Off in production; errors logged to file only

9.1 CSRF Token Implementation
// includes/csrf.php
function csrf_token() {
  if (empty($_SESSION["csrf_token"])) {
    $_SESSION["csrf_token"] = bin2hex(random_bytes(32));
  }
  return $_SESSION["csrf_token"];
}

function csrf_verify() {
  if (!hash_equals($_SESSION["csrf_token"], $_POST["csrf_token"] ?? "")) {
    http_response_code(403);
    die("Invalid CSRF token");
  }
}

// In every form:
<input type="hidden" name="csrf_token" value="<?= csrf_token() ?>">

A. PDO Database Singleton
// includes/db.php
function get_db(): PDO {
  static $pdo = null;
  if ($pdo === null) {
    $dsn = "mysql:host=".DB_HOST.";dbname=".DB_NAME.";charset=utf8mb4";
    $pdo = new PDO($dsn, DB_USER, DB_PASS, [
      PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
      PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
      PDO::ATTR_EMULATE_PREPARES   => false,
    ]);
  }
  return $pdo;
}

B. Output Escaping
// includes/functions.php
function e(string $str): string {
  return htmlspecialchars($str, ENT_QUOTES | ENT_HTML5, "UTF-8");
}

// Usage in templates:
<h1><?= e($product["name"]) ?></h1>

C. Settings Helper
function get_setting(string $key, string $default = ""): string {
  static $cache = [];
  if (!isset($cache[$key])) {
    $stmt = get_db()->prepare("SELECT setting_value FROM settings WHERE setting_key=?");
    $stmt->execute([$key]);
    $cache[$key] = $stmt->fetchColumn() ?: $default;
  }
  return $cache[$key];
}

// Usage:
$site_name = get_setting("site_name", "My Store");
